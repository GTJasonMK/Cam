# UI 问题整改落地（仅执行版）

【输入】
- review_report_path：UI 审查报告路径（或直接问题清单）。
- scope：页面/路由/组件范围（文件路径优先）。
- top_tasks：最多 3 个关键用户任务。
- acceptance_criteria：本轮完成标准（如 P0 全修、键盘可达、错误态可见、断点可用）。
- constraints：不可改动项（不换框架/不改后端/不改设计 token 等）。

【需要完成的功能】
- 基于 UI 审查报告执行整改，完成可直接合入的改动并给出验证证据。

【注意事项】
- 通用执行基线（必须）：先确认目标与验收（≤3 行）→扫描仓库事实→3-7 步执行（每步含影响面/验证/回滚）→回填证据；最多追问 3 个阻塞问题；每条 `acceptance_criteria` 必须映射证据；至少 1 条失败路径验证；按【输出】原样输出。
- 仅执行整改，不重复做全面 UI 审查；新发现问题写入风险补充。
- 优先级：P0（可用性/a11y/交互状态）→P1（一致性/布局稳健）→P2（视觉细节）。
- 优先复用现有组件与样式体系，避免一次性重写。
- 能跑就跑 lint/test/e2e/smoke 并留痕；不能跑给 `top_tasks` 的手工回归步骤。

【输出】
- 输入确认结果（审查来源、问题总数、纳入范围）。
- 执行计划（3-7 步，每步含影响面/验证/回滚点）。
- 变更清单（按文件：改动内容/原因/覆盖问题编号）。
- 验证命令与结果摘要（失败项单列）。
- 风险补充（未修项/原因/下一步）。
- 手工回归清单（围绕 `top_tasks`）。
- 验收映射表（acceptance_criteria → 改动点 → 验证证据）。
- 机器可读摘要（JSON）：{"status":"success|partial|blocked","scope":[],"changed_files":[],"verification":[],"unresolved":[],"next_actions":[]}。

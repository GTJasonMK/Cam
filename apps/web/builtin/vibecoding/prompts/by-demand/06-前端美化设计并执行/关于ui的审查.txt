# UI/UX 审查（仅审查版）

【输入】
- scope：页面/路由/组件范围（文件路径优先）。
- report_path：报告路径（默认 `.conversations/reports/ui-ux-review-report.md`）。
- top_tasks：最多 3 个关键用户任务。
- acceptance_criteria：可用性与体验目标（如键盘可达、错误提示清晰、点击热区达标）。
- constraints：不可改动项（不换 UI 框架/不改后端/保持现有组件库等）。

【需要完成的功能】
- 仅做 UI/UX 审查（不改代码），输出问题分级与整改方案。
- （可留空）platform/device、参考截图/URL/Figma。

【注意事项】
- 通用审查基线（必须）：先扫描事实→再分级问题→最后给路线图；最多追问 3 个阻塞问题；每条 `acceptance_criteria` 必须有证据；至少覆盖 3 个高风险边界；按【输出】原样输出。
- 默认从仓库推断入口、组件树、样式方案和验证命令，不反问项目基础信息。
- 优先级：先 P0（可用性/a11y/loading/empty/error/focus），再 P1（一致性/布局稳健），最后 P2（视觉细节）。
- 每条问题必须给定位证据（组件/选择器/交互路径/截图位置）和可执行修正建议（尽量含数值）。
- 能跑就跑相关检查命令并留痕；不能跑写阻塞原因与可信度影响。
- 报告必须落盘到 `report_path`，并在结尾回写路径与摘要。

【输出】
- 扫描结论（入口、样式体系、关键路径、断点/设备策略）。
- 问题清单（P0/P1/P2，字段：位置/证据/修正方案/验证方式）。
- 整改优先级路线图（P0→P1→P2，含收益/风险/工作量）。
- 验证清单（已执行命令 + 结果摘要；未执行项手工步骤）。
- 审查报告（已保存路径 + 报告摘要）。
- 审查结论（可发布/需修复后复审/阻塞项）。
- 验收映射表（acceptance_criteria → 问题分级/证据/修复建议）。
- 机器可读摘要（JSON）：{"status":"success|partial|blocked","scope":[],"findings":[],"evidence":[],"priority_plan":[],"next_actions":[]}。

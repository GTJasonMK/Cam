# 借鉴落地回归审查（仅审查版）

【输入】
- scope：审查范围（模块/目录/功能链路）。
- report_path：报告路径（默认 `.conversations/reports/cross-project-adoption-review-report.md`）。
- execution_handoff_path：执行步骤交接文件路径（如 `.conversations/pipeline-handoffs/<pipeline_id>/step-handoff.json`）。
- acceptance_criteria：至少包含“借鉴项完成度、风险清单、复审结论、下一步建议”。
- constraints：审查深度、时间预算、是否允许运行验证命令。
- （可留空）handoff_output_dir：默认 `.conversations/pipeline-handoffs/<pipeline_id>/`。

【需要完成的功能】
- 仅做借鉴落地后的回归审查：确认借鉴项有效且当前项目关键能力不回退（不改代码）。

【注意事项】
- 通用审查基线（必须）：先扫描事实→再分级问题→最后给结论；最多追问 3 个阻塞问题；每条 `acceptance_criteria` 必须有证据；按【输出】原样输出。
- 严格单一职责：仅审查不改代码。
- 必须读取 `execution_handoff_path`，并逐条核对“借鉴项 → 实际改动 → 验证结果”。
- 重点核对：关键业务路径、兼容性、性能、稳定性、维护复杂度变化。
- 报告必须落盘到 `report_path`；如提供 `handoff_output_dir`，结束时写 `step-summary.md` 与 `step-handoff.json`。

【输出】
- 扫描结论（审查范围、输入来源、可用验证命令）。
- 问题清单（P0/P1/P2：位置/影响/证据/修复建议/验证方式）。
- 借鉴项完成度（借鉴项 → 完成状态 → 证据）。
- 回归风险清单（功能/性能/兼容性/维护成本）。
- 验证清单（已执行命令 + 结果摘要；未执行项手工步骤）。
- 审查报告（已保存路径 + 报告摘要）。
- 复审结论（可合入/需修复后复审/阻塞）。
- 流水线交接文件（若提供 `handoff_output_dir`：`step-summary.md` + `step-handoff.json`）。
- 验收映射表（acceptance_criteria → 证据 → 结论）。
- 机器可读摘要（JSON）：{"status":"success|partial|blocked","scope":[],"adoption_status":[],"regression_risks":[],"next_actions":[]}。
